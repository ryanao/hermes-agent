# 整理微信文件传输助手

## When to Use

当用户说"帮我整理一下文件传输助手""帮我理理微信里堆的东西""把这段微信记录分类"时触发。
或者用户直接调用 `/wechat-ft-organizer`。

## Inputs

用户会提供以下之一：

1. **直接粘贴** 一段从微信复制的消息流（最常见）
2. 一个 **文件夹路径**（微信 PC 版的文件传输助手导出目录）
3. 一段时间内的 **聊天记录导出**（纯文本/HTML）

如果用户只说"帮我整理一下"但没给内容，**先问他想整理什么**——直接粘贴、还是给一个导出文件的路径。

## Procedure

### 1. 理解输入

- 如果是粘贴的文字：直接分析
- 如果是文件路径：用 `read_file` / `list_dir` 先看目录结构，识别文件类型
- 如果看不懂格式（微信导出的 HTML/txt 各家工具不同），先取前 20 行让用户确认

### 2. 分类

把每一条内容归入下面 5 类，归不进去的放到"其他"：

| 类别 | 判定标准 |
|------|---------|
| 📎 **待读链接** | URL / 公众号文章 / 视频链接 |
| 📄 **文档与文件** | PDF / Word / Excel / 图片文件名 / 附件引用 |
| 💡 **想法碎片** | 一两句随手写的思考、灵感、观察 |
| ✅ **待办事项** | 以"要""需要""别忘""记得""todo"等开头，或带时间/日期 |
| 🖼 **图片与媒体** | 图片、语音、视频引用 |
| 🗑 **其他** | 以上都不像，或者明显是临时复制的废弃内容 |

### 3. 输出

按下面的格式输出（使用 Markdown，保留原始内容片段，长内容截断到一行摘要）：

```markdown
# 文件传输助手整理结果（共 N 条）

## 📎 待读链接 (X)
- [标题/首句摘要] — <URL>
- ...

## 📄 文档与文件 (X)
- 文件名 — 简述内容（如果能推断）
- ...

## 💡 想法碎片 (X)
> 原文一句话引用
> ...

## ✅ 待办事项 (X)
- [ ] 待办内容 — （时间：xxx，如果有）
- ...

## 🖼 图片与媒体 (X)
- 图片/语音引用 — 上下文
- ...

## 🗑 其他 (X)
- ...

## 建议下一步
- 待读链接中优先级高的 3 条：...
- 应该立即处理的待办：...
- 可以直接删除的：...
```

### 4. 询问保存

整理完后问用户：

> 要把这份整理结果保存到 `~/notes/wechat-ft/YYYY-MM-DD.md` 吗？

如果用户同意，用 `write_file` 保存到 `~/notes/wechat-ft/` 目录（按日期命名），没有目录就先 `mkdir`。

## Tips

- **不要丢内容**：哪怕判定不清，也归到"其他"而不是删掉
- **长链接要保留原始 URL**，不要用 markdown 的 `[text](url)` 折叠
- **语音/图片**如果只有占位符（比如 "[图片]"），如实标记，不要瞎猜内容
- **时间信息**很宝贵，能从消息中提取就提取（"昨天提的"、"下周三的会"）
- 判断"待办"时保守一点：模糊的就归"想法"，不要强行把每句话都变成任务
- 如果内容超过 200 条，**先给个分类计数总览**，再问用户是否要详细列出

## 示例触发

**输入**（用户粘贴的一段）：

```
2024/4/20 9:30
https://mp.weixin.qq.com/s/abc123 这篇讲 LLM agent 的不错
记得周五前交那个报告
[图片]
餐厅地址: 朝阳区xx路123号
别忘了问张总下季度预算
python 装饰器本质是闭包 + 语法糖
```

**期望输出**：

- 📎 1 条（公众号链接）
- ✅ 2 条（交报告 / 问张总预算）
- 💡 1 条（装饰器笔记）
- 🖼 1 条（图片占位符）
- 🗑 1 条（餐厅地址，可能是临时复制，让用户判断）
